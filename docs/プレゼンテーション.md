# Tokyo Heat Safe Outing — プレゼンテーション

## タイトル

Tokyo Heat Safe Outing — 暑熱対策付きおでかけ支援プラットフォーム

## 1. サービス概要

- 一文サマリ: 東京都オープンデータを用いて、屋外行動の暑熱リスクを可視化・回避する経路・施設提案を行う市民向けアプリ。
- コア機能: 周辺の暑さ低減に寄与する施設（給水所、日陰、公園、避難所等）を可視化、熱中症リスクスコア付きの安全ルート提示、行動ログによるパーソナライズ。

## 2. 行政課題の定義（問題提起）

- 背景: 都市部の気温上昇とヒートアイランド現象により、外出時の熱中症リスクが増加している。
- 現状の課題:
  - 市民は屋外で瞬時に近くの安全な場所（冷却・給水・避難）を把握しづらい。
  - 行政は設置済み施設の利用促進や最適配置の評価が十分できていない。
- 影響: 高齢者・子ども・屋外労働者の健康被害増加、救急搬送の増加による医療負荷。

## 3. 使用するオープンデータと取り回し

- データソース（東京都オープンデータカタログ）:
  - 施設データ: `data/facilities.json`（給水所、トイレ、避難所、保健センターなど）
  - 施設種別: `data/facility-types.json`
  - （将来的）気象データAPI（気温・湿度・熱中症危険度） — 都や気象庁API連携想定
- データ処理フロー（簡潔）:
  1. 施設データをジオインデックス化（R-tree / GeoJSON）
  2. 現在地 + 時刻 + 気象データで「局所熱リスク」を算出（湿球黒球温度や不快指数を簡易計算）
  3. リアルタイムで近隣施設の冷却能力・開放状況をスコア化し、ルート最適化へ入力

## 4. 技術的ロジック（審査で問われる“有効活用”と“技術的特異性”の説明）

- リスクスコア計算
  - 入力: 気温、湿度、日射情報（推定）、歩行速度、日陰率（地物情報）
  - 出力: 0–100の熱中症リスクスコア／区間
  - ロジック要旨: 簡易WBGT相当値の推定 + 歩行時間加重 + 施設での短時間リセット効果を反映
- ルート最適化（技術的特徴）
  - 拡張最短経路：距離だけでなく「累積熱リスク」をコストに組込み（Dijkstra 拡張）
  - ノンラインナップ: 途中で給水/休憩を挟むよう経路分割（Waypoint挿入）
  - 高度化案: OSRMなどのルーティングエンジンと連携してリアルタイム経路を作成（`OSRM.md` 参照）
- 空間検索・可視化
  - クライアント側でベクトルタイルまたはGeoJSONで描画、ヒートマップレイヤーで熱リスク表示
  - オフラインキャッシュ（小エリア）で外出先でも機能維持
- データ活用の独自性
  - 東京都施設データを「行動支援」のコスト関数に直接組み込む点が差別化要素
  - 個人行動ログを匿名集約し、行政向けに熱ストレス多発地点の可視化を提供

## 5. サービス実現性（事業主体・運用モデル）

- 想定事業主体
  - 地方自治体（区市町村）主導、またはNPO / 民間と連携した公民連携モデル
- 運用フロー
  - データは自治体の公開データを定期同期。災害時は優先的に運用モード切替（避難所情報の更新）
  - 施設管理者からのAPIや簡易フォームで「臨時閉鎖」「冷房あり」等を更新可能
- ビジネスモデル（継続性）
  - B2G: 行政への分析レポート提供・インフラ最適配置コンサルティング
  - B2C: プレミアム機能（月次のパーソナライズ通知・企業スポンサー枠）
  - 協賛: スマホメーカー・飲料メーカーによるスポンサー表示で無料提供を維持

## 6. UX と新しい体験（利用者価値）

- リアルタイム安全ナビ: 熱リスクが低いルートを案内し、途中で最短の給水ポイントへ立ち寄らせる
- パーソナライズ: 年齢層や持病設定で閾値を調整、音声案内や大きな文字モードで高齢者対応
- コミュニティ機能: 利用者の匿名報告で「混雑している避難所」や「臨時給水ポイント」を共有
- 行政向けダッシュボード: 熱ストレスホットスポットの時間変化を可視化し、資源配分を支援

## 7. 技術スタック（短く）

- フロント: HTML/CSS/JavaScript（既存の `index.html`, `main.js`, `styles.css` を拡張）
- ルーティング: OSRM / GraphHopper など（`OSRM.md` に連携案）
- サーバ/API: Node.js（軽量API）、キャッシュ層にRedis（運用時）
- データ: GeoJSON, R-tree インデックス、東京都オープンデータ

## 8. セキュリティ・プライバシー

- 位置情報は端末で計算し、サーバへ送らない設計をデフォルトに（匿名化が必要な場合のみ集約データを送信）
- GDPR/個人情報保護に沿った同意フローと設定画面

## 9. KPI と成功指標

- 月間アクティブユーザー数（MAU）
- アプリ内での「安全ルート選択率」
- 行政向け：ヒートストレス多発地点の検出精度（既存通報との照合）
- ユーザー反応（満足度、通知による事故回避報告）

## 10. 今後の展開（機能不足と優先実装案）

優先度高

- リアルタイム気象連携（気象庁・都のAPI）→ リスク推定精度向上
- ルーティングエンジン接続（OSRMサーバ立ち上げ・API統合）
- 施設ステータス更新UI（臨時閉鎖・開放情報を施設管理者が更新）
- 高齢者向けアクセシビリティ設定（音声・拡大文字・単純表示）

中期

- 機械学習による利用パターン学習（ユーザーの行動で最適閾値調整）
- 行政向け自動レポート生成（CSV/PDF）
- 多言語対応（英語・中国語・韓国語）

長期 / 研究的

- 屋外日陰率推定（センサ・衛星データやLIDARを用いた都市気候解析）
- ウェアラブル連携（心拍等のバイタルで個人リスク最適化）

## 11. 実装・検証計画（短期スプリント案）

- 1週目: データパイプライン整備、`data/*.json` の読み込みと地図表示
- 2週目: 熱リスクスコア算出モジュール実装 + 単純ルート最適化（距離＋リスク）
- 3週目: モバイルUI調整、ユーザーテスト（5名のエンドユーザー）
- 4週目: OSRM連携、行政レポートのプロトタイプ作成

## 12. 簡単な技術的契約（Inputs / Outputs / 成功基準）

- Inputs: 現在地座標、時刻、（気象データ）、`data/facilities.json`
- Outputs: ユーザーに提示される「安全ルート」と「推定熱リスクスコア」、近隣施設リスト
- 成功基準: 実ユーザーテストで「提示されたルートを安全だと感じた」評価が70%以上

## 13. 想定されるエッジケース（3–5）

- 施設情報が古く閉鎖中 → 施設ステータスの更新UIとユーザーレポートでカバー
- ネットワーク切断時の位置把握 → ローカルキャッシュで基本機能維持
- 高密度都市部での多施設選択のオーバーヘッド → クラスタリングと優先スコアで絞り込み
- 個人差（高齢者等）による耐熱性違い → プロファイルで閾値を個別化

## 15. まとめ（短く）

- 本サービスは東京都の公開施設データを直接「個人の行動支援」に落とし込み、熱中症リスクを低減する点で行政課題に直結している。技術的にはルート最適化へのリスクコスト導入、リアルタイム気象連携、自治体向け分析を組み合わせることで実現性と独自性を両立する。
